rOpenSci Statistical Software Peer Review

Chapter 4 Guide for Editors

4.1 Editor-in-Chief

As for rOpenSci’s current peer-review system, packages are submitted directly to the ropensci/software-review repository on GitHub, with submissions handled initially by the rotating Editor-in-Chief (EiC). Statistical software is, however, handled differently from the first moment of submission. Statistical submissions are identified as such on the submission form, and this automatically generates a response from our ropensci-review-bot that indicates whether a submission may proceed, based on whether it passes all of the automatic checks listed in the following sub-section.

A positive bot response is sufficient for the EiC to proceed directly to delegate a handling editor without further consideration. Additional effort by the EiC will only be required in “edge cases” where a package may be unable to pass one of the checks. In these cases, submitting authors must explain why these checks may fail, and the EiC must then determine whether these failures are acceptable.

There only additional responsibilities for the EiC prior to delegating handling editors are to check the following single item:

  • The categories nominated by the submitting authors are appropriate for this package

And to choose one the following two items:

  • The package does not fit within any additional categories of statistical software.
  • The package could potentially be described by the following additional categories of statistical software:
    • <list categories here>

4.1.1 Bot checks executed on initial submission

The checks prompted by an initial submission, and required for the EiC to immediately proceed to delegate a handling editors, are:

  • srr_stats_pre_submit() must confirm that all standards have been addressed, and that a “Life Cycle Statement” has been included within the CONTRIBUTING.md document.
  • R CMD check must generate no errors, and all warnings must be explained.
  • All tests must pass
  • Test coverage must be at least 70%
  • The package must use roxygen2 to generate documentation.
  • No function should have cyclomatic complexity of 15 or greater
  • The package must contain at least one vignette
  • The package must be documented on an associated website

4.2 Handling Editor

The list of steps to be followed by the Handling Editor is derived from the equivalent steps in the Dev Guide, yet slightly different here both because Statistical Software review necessitates a couple of additional checks, and because not all of the general checks are applicable.

Checks from Dev Guide to be applied to Statistical Software:

  • Check against policies for fit and overlap. Initiate discussion via Slack #software-review channel if needed for edge cases that haven’t been caught by previous checks by the EiC. If reject, see this section about how to respond.
  • For packages needing continuous integration on multiple platforms (cf criteria in this section of the CI chapter) make sure the package gets tested on multiple platforms (having the package built on several operating systems via GitHub Actions for instance).
  • Wherever possible when asking for changes, direct authors to automatic tools such as usethis and styler, and to online resources (sections of this guide, sections of the R packages book) to make your feedback easier to use. Example of editor’s checks.
  • Ideally, the remarks you make should be tackled before reviewers start reviewing.
  • If initial checks show major gaps, request changes before assigning reviewers. If the author mentions changes might take time, apply the holding label.
  • If the package raises a new issue for rOpenSci policy, start a conversation in Slack or open a discussion on the rOpenSci forum to discuss it with other editors (example of policy discussion).

Additional Checks

  • Appropriately resolve any issues raised during initial processing by EiC.
  • Run autotest_package() on a local version of the source code and ensure no errors are raised.
  • Ensure that the submitting authors understand the requirements for their package to meet their stated grade of badge (gold, silver, bronze), and for silver in particular clarify which of the four aspects listed in the “Guide for Authors” section on silver badges the authors intend to fulfil.

Beyond that, Handling Editors follow the procedures given in the Dev Guide, particularly in regard to looking for and assigning reviewers.
